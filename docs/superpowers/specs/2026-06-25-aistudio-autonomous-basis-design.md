# AI Studio Demo 自治化基础设计

## 背景
当前仓库是一个 AI Studio demo 原型，已有 `3D 地球`、`AI Elements`、`Supabase` 和 `.env` 配置入口，但还不是可长期维护的产品骨架。参考项目 `pposintel` 证明了：先建立数据契约、配置边界和文档记录，再做 UI 和自动化，后续才有复制和自治的基础。

## 目标
第一轮只做一个可落地的“双层结构”：

1. 产品层：当前应用可稳定运行、可手动维护更新、可在单台服务器上恢复。
2. 自治层：系统具备最小自检、自修、发布、回滚能力，为后续复制新实例预留接口。

## 设计原则
- 保持单体应用形态，不提前引入多服务编排。
- Supabase 只承担配置和读模型，不把 demo 改造成重后端平台。
- `.env` 作为统一配置入口，LLM 与 Supabase 都必须可替换、可恢复。
- 自治优先级高于扩展性：先保证能活，再保证能复制。

## 架构草案
### 产品层
- 前端继续沿用当前原型的页面结构与交互方式。
- `3D 地球` 只负责导航、态势和数据可视化，不承担业务写入。
- `AI Elements` 只负责对话、输入、摘要、解释与操作面板。
- Supabase 作为最小持久化层，保存可恢复配置、schema 和必要的业务读数据。

### 自治层
- `startup-check`：启动时检查 `.env`、Supabase 连接、LLM provider、必要文件和端口。
- `health-check`：运行中检查页面、核心接口和基础数据是否可用。
- `update-check`：执行 git 驱动更新，完成安装、构建和 smoke test。
- `rollback`：任一步失败时回退到上一个稳定 commit。

## 未来复制能力
复制不是第一轮主目标，但要提前留接口：
- 新实例初始化脚本
- `.env` 模板与校验
- Supabase schema/seed 初始化
- git 基线恢复

## 明确不做
- 多实例编排
- 镜像仓库发布流
- 插件市场
- 完整自复制控制面
- 复杂模型路由平台

## 验收标准
- 单台服务器可以启动并维持运行。
- 配置错误可以被检测出来，而不是静默失败。
- 更新失败可以回滚到上一个稳定版本。
- 文档里能清楚区分“当前可用能力”和“未来自治能力”。
- 后续实现计划可以直接基于这份设计拆分任务。
